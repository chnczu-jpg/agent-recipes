# Agent Recipes Competitive Standard

Date: 2026-07-10

## 大白话目标

“比 Cass、Remnic 和 TencentDB Agent Memory 加起来更强”不能理解成把它们所有功能都抄一遍。
那样只会得到第三个更胖的系统。

这里的“更强”只认一个结果：

> Agent 更容易找到真正适用的经验，敢于说没有匹配，执行时受菜谱约束，失败后能修正，
> 而且整套核心仍然轻、可检查、可迁移、可卸载。

Cass 擅长搜索大量历史会话；Remnic 擅长广泛记忆、召回和生命周期；TencentDB Agent Memory
擅长分层会话记忆、上下文压缩和宿主恢复流水线；Agent Recipes
要做的是更严格的经验治理和执行闭环。搜索与记忆可以接可选引擎，不能重新塞进核心。

三仓库逐项真值和当前阻断项见 `THREE_REPO_COMPETITIVE_SCORECARD.md` 与同名 JSON。
统一规则是：每一项取三个仓库里最强的能力，任何硬短板都不能被其他项目的得分抵消。

## 什么时候才允许说更强

下面九道门全部有真实证据后，才能说在“Agent 安全复用经验”这个收窄目标上超过
三个仓库的最强组合。少一项都只能说局部领先。

| 硬门 | 必须做到 | 当前状态 |
|---|---|---|
| 人工转正 | 候选绝不自动变正式菜谱 | 已过本地测试 |
| 不乱套 | 弱匹配必须 no-match，执行前锁版本和 hash | 已过本地测试 |
| 失败会学习 | 纠偏、课程、产物、成功链能自动产候选，人只审核 | 固定 19 项目/1112 卡本地大样本门已过；新领域仍须复测 |
| 真正减少犯错 | fresh agent 在新任务里稳定比无菜谱更少犯错 | 小样本，未过生产级 |
| 生命周期可靠 | 退役、防复活、成功/失败/未知归因、坏账本停写 | 已过本地测试 |
| 证据可信 | 来源可追、坏证据隔离、敏感信息不落盘、缺证据敢停 | 已过本地测试 |
| 记忆够用 | 可选记忆/搜索能召回有用候选，但不能改正式事实 | v0 已过，质量规模未过 |
| 工程成熟 | 托管 CI、版本历史、升级、第二台机器、回滚都真实跑通 | 未过 |
| 更轻 | 核心零运行依赖、零必需外部服务、wheel 不超过 256 KiB | 当前已过 |

## 轻量硬预算

机器真值在 `ENGINEERING_BUDGET.json`。当前门槛：

- 最小核心运行依赖：0；
- 必需外部服务：0；
- wheel：不超过 256 KiB；
- `core.py`：不能超过本次拆分后的 15610 行，只能继续下降；
- 全包 Python 源码：不超过 950000 字节；
- 本机冷启动中位数：不超过 150 ms。

这些是上限，不是鼓励把空间用满。要提高预算，必须同时给出真实用户收益、基准结果和
为什么不能放进可选模块。

## 比它们强的正确方法

1. 核心只保留治理：事件账本、候选转正、no-match、锁、结果归因、生命周期、doctor。
2. 资料拆解、LLM、embedding、图谱、OCR、ASR 都是可选模块，坏了不拖垮核心。
3. 搜索广度不重复造轮子。允许接 MIT 且通过审计的引擎，但召回结果永远只是候选证据。
4. 每项“更强”都要有 A/B：无菜谱、使用菜谱、重复错误、时间、人工介入、最终质量。
5. 第三个技能仍不能比第一个明显更快，就说明复用失败，停止加功能，先修主链。

## 明确不做

- 不在核心重造通用会话搜索、TUI、SSH 同步、向量数据库或图数据库；
- 不让 LLM、embedding、Cognee、Graphiti 或未来 Remnic adapter 写正式菜谱；
- 不复制受限 Cass 源码；
- 不为了工具数量增加浅模块和 MCP 命令；
- 不把测试通过、召回命中或候选生成冒充真实任务质量。

## 下一条实现顺序

1. 扩大 fresh Codex 在新任务里的无菜谱/有菜谱行为对照，证明不是只会产候选；
2. 用同一固定 cohort 比较可选搜索/记忆 adapter 的准确率和误召回；
3. 建立版本历史、托管 CI 和第二台机器复现；
4. 九道门全部过后，再按三仓库统一总表做正式结论。

当前有本地证据的门是 `6/9`。这只表示“失败会学习”在固定 SampleProject cohort 上达到本地门槛，
不表示任何新资料都能稳定学对，也不表示整体已经超过三个仓库。统一总表当前明确判定：
收窄目标只到局部领先，全功能目标明确没有超过。
